# 架构升级方案

## 1. 背景与目标

当前仿真器已经具备以下基本链路：

`trace/onnx -> Model/Operation -> Core -> ICNT -> DRAM/SSD -> Core`

但为了支持后续生成式推荐模型（GR）与软硬件协同设计，现有架构存在三个关键问题：

1. `SSD` 采用与 `DRAM` 同步的小步推进方式，真实参数下（例如 `pg_rd_lat = 40us`）会在宿主机上产生大量空转循环。
2. `Simulator` 当前直接按物理地址把请求分流到 `DRAM/SSD`，缺少统一的存储控制平面，无法支撑后续的 `DRAM <-> SSD` 迁移、embedding 分层存储与动态调度。
3. 缺少独立的存储 benchmark 能力，无法系统性输出 `DRAM/SSD` 在不同访问大小、不同 burst 深度下的延迟、吞吐与排队效应。

本轮升级按如下顺序实施：

1. 先修改 `SSD` 的集成方式，消除真实 SSD 参数带来的低效推进问题。
2. 再引入“总控制器/存储控制平面”，把 `NPU <-> DRAM/SSD` 与 `DRAM <-> SSD` 统一到同一控制路径。
3. 最后实现 `mem_benchmark`，生成 CSV 和图表。

## 2. 总体设计原则

### 2.1 顶层采用混合推进

不把整个仿真器重写成纯事件驱动，而采用“组件内部自选推进方式 + 顶层按下一事件时间快进”的混合方案：

- `Core / Scheduler / NoC` 仍保留现有周期级行为。
- `Ramulator2` 内部保持时钟驱动。
- `SSD` 内部改为事件驱动。
- `Simulator` 顶层不再假设所有组件都必须按固定小步一起前进，而是允许对长延迟存储事件进行时间快进。

### 2.2 区分逻辑事务与物理微请求

后续控制器、benchmark、embedding 都不能再只依赖“物理地址 + 32B 请求”的语义，需要区分：

- 逻辑事务：例如一次 4KB embedding row 读取、一次 64KB 迁移、一次 benchmark 宏访问。
- 物理微请求：为适配 DRAM/NoC/SSD 设备而拆分出来的底层请求。

### 2.3 存储控制平面独立

后续一切与 `tiered memory` 相关的能力统一由控制平面承接，而不是散落在 `Simulator / Tensor / Operation` 中：

- NPU 发起的读写
- DRAM/SSD 地址路由
- DRAM <-> SSD 数据交换
- benchmark 注入请求
- 未来 embedding placement / prefetch / eviction 策略

## 3. 分阶段修改方案

## 3.1 第一阶段：修改 SSD 集成方式

### 3.1.1 目标

- 把当前 `SSD` 从“每个 DRAM tick 都调用一次 `cycle()`”改为“基于绝对物理时间的完成事件”。
- 为顶层提供 `next_event_time_ps()` 风格接口。
- 保持当前功能行为不变：仍支持 NPU 请求按地址进入 SSD，并返回到 Core。
- 在不重写 DRAM 和 Core 的前提下，先把 SSD 从顶层小步推进中解耦。

### 3.1.2 核心改动

#### A. 重构 `Ssd`

新增/调整接口：

- `void push(MemoryAccess* req, uint64_t now_ps)`
- `void advance_to(uint64_t now_ps)`
- `bool has_pending() const`
- `uint64_t next_event_time_ps() const`
- `bool has_ready_response() const`
- `MemoryAccess* top_ready_response()`
- `void pop_ready_response()`

内部不再用 “本地 cycle + finish_cycle” 表示请求完成时间，而是直接记录绝对时间：

- `finish_time_ps`
- `issue_time_ps`
- `mem_enter_time_ps`
- `mem_finish_time_ps`

#### B. 调整 `Simulator`

把当前：

- `if (_cycle_mask & DRAM_MASK) { _dram->cycle(); if (_ssd) _ssd->cycle(); }`

改为：

- `DRAM` 保持原样推进。
- `SSD` 只在以下两种情况下被访问：
  - 新请求进入 SSD
  - 全局时间推进到其下一完成事件时间之后

#### C. 增加阶段时间戳

在 `MemoryAccess` 上补齐阶段时间信息，至少包括：

- `issue_time_ps`
- `mem_enter_time_ps`
- `mem_finish_time_ps`
- `return_time_ps`
- `logical_size_bytes`
- `macro_request_id`
- `target_medium`

### 3.1.3 验证目标

- 原有 ONNX / trace 路径能正常运行。
- SSD 请求仍可正确返回。
- 使用真实 SSD 参数时，单个请求不再依赖数万次 `SSD::cycle()` 空转。

## 3.2 第二阶段：总控制器 / 存储控制平面升级

### 3.2.1 目标

引入统一的 `StorageController`（或 `TieredMemoryController`），把当前散落在 `Simulator` 中的存储逻辑收敛到控制平面，为后续 embedding / DRAM <-> SSD 迁移做准备。

### 3.2.2 控制平面职责

#### A. 统一接收逻辑存储请求

来源包括：

- NPU Core 产生的读写请求
- benchmark front-end 注入的请求
- 后续控制器内部的 migration 请求

#### B. 统一做介质选择与设备派发

根据以下信息决定流向：

- 物理地址区间
- placement 元数据
- 控制策略
- 请求类型（普通读写、预取、迁移、benchmark）

#### C. 统一回收设备返回事件

控制平面负责收集：

- DRAM 返回
- SSD 返回
- migration 完成

并决定如何把结果送回：

- NPU Core
- 控制器内部状态机
- benchmark recorder

#### D. 为后续 embedding 与分层存储保留扩展点

控制平面需要预留这些接口：

- `submit_npu_request(...)`
- `submit_benchmark_request(...)`
- `submit_migration_request(...)`
- `register_placement(...)`
- `query_placement(...)`

### 3.2.3 关键数据结构

#### `MemoryRequest`

逻辑请求层抽象，包含：

- 请求类型
- 来源/目的
- medium hint
- logical object id / tensor name
- physical address
- size
- rw
- timing fields

#### `MigrationRequest`

描述 `DRAM <-> SSD` 交换：

- `src_medium`
- `dst_medium`
- `src_addr`
- `dst_addr`
- `bytes`
- `priority`

#### `PlacementEntry`

为后续 embedding 分层存储保留元数据：

- object name / table id
- range / shard / row range
- medium
- capacity class
- policy tags

### 3.2.4 与 embedding 的关系

当前阶段不实现完整 `Embedding op`，但控制平面必须预留能够承接 embedding 的接口。

后续 embedding 设计原则：

- `Embedding` 不再假设整张表只有一个物理基地址。
- `Embedding` 向控制平面提交“查表意图”，而不是直接拼装最终物理地址。
- 控制平面根据 placement 决定：
  - DRAM 命中直接返回
  - SSD 直读
  - 或先触发 SSD -> DRAM 迁移

### 3.2.5 验证目标

- 原有 NPU 请求经过 `StorageController` 后行为与之前一致。
- 新增 migration 接口可被调用并完成 DRAM <-> SSD copy。
- benchmark 和未来 embedding 可复用同一控制器接口。

## 3.3 第三阶段：mem_benchmark

### 3.3.1 目标

支持对 `DRAM/SSD` 做独立存储 benchmark，测试内容至少包括：

- 访问大小：`512B / 1KB / 2KB / 4KB / 8KB / 16KB / 32KB / 64KB`
- 连续请求数：`1 / 2 / 4 / 8 / 16 / 32 / 64 / 128`
- 介质：`DRAM / SSD`
- 操作：`read / write`

### 3.3.2 benchmark 形式

新增 `mem_bench` 模式，而不是强行复用现有 ONNX/trace/full-core 执行流。

原因：

- benchmark 目标是测存储，不是测 compute。
- 现有 operation 会引入计算、tiling、buffer 依赖，不利于控制变量。
- 控制平面已经能够直接接收逻辑存储请求，适合作为 benchmark 注入口。

### 3.3.3 指标定义

对每个 case 输出：

- `first_response_latency_ns`
- `last_response_latency_ns`
- `avg_macro_latency_ns`
- `avg_micro_latency_ns`
- `p50/p95/p99`
- `total_bytes`
- `effective_bandwidth_GBps`

### 3.3.4 CSV 输出

至少生成两份：

#### `summary.csv`

每个 case 一行：

- `medium`
- `rw`
- `access_size_bytes`
- `burst_count`
- `micro_request_count`
- `first_response_latency_ns`
- `last_response_latency_ns`
- `avg_macro_latency_ns`
- `avg_micro_latency_ns`
- `bandwidth_GBps`

#### `detail.csv`

每个微请求一行：

- `macro_request_id`
- `micro_request_id`
- `medium`
- `rw`
- `address`
- `size`
- `issue_time_ps`
- `mem_enter_time_ps`
- `mem_finish_time_ps`
- `return_time_ps`
- `channel`
- `lun`

### 3.3.5 图表输出

新增绘图脚本，读取 `summary.csv`，输出：

- DRAM read latency 曲线图
- SSD read latency 曲线图
- DRAM write latency 曲线图
- SSD write latency 曲线图

建议：

- 横轴：`access_size_bytes`
- 纵轴：`last_response_latency_ns`
- 每条曲线对应一个 `burst_count`
- 使用对数刻度显示横轴

### 3.3.6 验证目标

- benchmark 可以在同一配置文件下同时输出 DRAM/SSD 结果。
- CSV 可直接被后处理脚本读取。
- 图表可自动生成。

## 6. 本轮实施边界

本轮代码实施以“把架构搭起来并能工作”为主，以下内容作为后续扩展项：

- 完整 `Embedding` operation
- embedding row-level placement / hot-cold 策略
- 用户负载特征驱动的在线调度策略
- DRAM/SSD 更复杂的 QoS / priority / prefetch 策略

本轮需要确保的核心交付：

1. SSD 集成方式已升级，真实参数不会导致低效的小步空转。
2. 存储控制平面已建立，后续 `DRAM <-> SSD` 与 embedding 扩展有统一入口。
3. mem_benchmark 可输出结构化 CSV 与图表。
