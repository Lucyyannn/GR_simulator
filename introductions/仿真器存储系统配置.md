# 仿真器存储系统配置

本文整理当前仿真器中片上 `SRAM`、片外 `HBM`、`DDR`、`SSD` 的硬件配置入口、默认存储策略，以及常见调整方法。本文对应的主配置样例是 [configs/systolic_ws_128x128_c4_simple_noc_tpuv4_half_ramulator2_ssd.json](/home/yihan.lu/GR_simulator/configs/systolic_ws_128x128_c4_simple_noc_tpuv4_half_ramulator2_ssd.json)。

## 1. 总体结构

当前仿真器的存储层级为：

- 片上：每个 `core` 自带 `SRAM` / `accum SRAM`
- 近端片外：`HBM`，由 `ramulator2 + HBM2.yaml` 驱动
- 远端主存：`DDR`，由 `ramulator2 + DDR4.yaml` 驱动
- 大容量后端：`SSD`，由 FEMU BlackBoxSSD 的事件驱动接口驱动

地址空间按介质分段：

- `HBM`: 从 `hbm.address_base` 开始，容量 `hbm.size_gb`
- `DDR`: 从 `ddr.address_base` 开始，容量 `ddr.size_gb`
- `SSD`: 从 `ssd.address_base` 开始，容量 `ssd.capacity_bytes`

当前默认样例中使用：

- `HBM` 基址 `0x0`，容量 `16 GB`
- `DDR` 基址 `0x400000000`，容量 `16 GB`
- `SSD` 基址 `0x800000000`，容量 `1 TB`

因此默认地址划分为：

- `0x000000000` ~ `0x3ffffffff` -> `HBM`
- `0x400000000` ~ `0x7ffffffff` -> `DDR`
- `0x800000000` 之后 -> `SSD`

## 2. 片上 SRAM

片上 `SRAM` 配置位于 `core_config.core_x` 中：

- `spad_size`: 主 scratchpad 容量，单位 `KB`
- `accum_spad_size`: 累加缓冲容量，单位 `KB`
- `sram_width`: 片上搬运宽度，当前样例为 `32 B`

这些参数控制的是 `core` 内部的 tile 驻留与片上数据复用，不参与外部地址空间划分。

当前样例中每个 core 为：

- `spad_size = 16384 KB`
- `accum_spad_size = 4096 KB`
- `sram_width = 32 B`

## 3. HBM 配置

`HBM` 配置入口：

- 兼容旧字段：`dram_*`
- 新字段：`hbm { ... }`

当前建议以后只改 `hbm` 块，旧 `dram_*` 仅用于兼容部分旧路径和日志。

关键字段：

- `hbm.enabled`
- `hbm.freq`
- `hbm.channels`
- `hbm.req_size`
- `hbm.nbl`
- `hbm.size_gb`
- `hbm.address_base`
- `hbm.config_path`

当前样例：

- `freq = 1200 MHz`
- `channels = 16`
- `req_size = 32 B`
- `nbl = 1`
- `size_gb = 16`
- `config_path = ../configs/ramulator2_configs/HBM2.yaml`

说明：

- `req_size` 是 NPU 发往外部存储系统的原子粒度，也是 `HBM` wrapper 的基础粒度
- `channels` 同时决定 NPU 直连 memory port 的数量
- `HBM` 是默认的近端工作集介质，算子性能模型仍主要按 HBM 带宽估算

## 4. DDR 配置

`DDR` 通过单独的 `ramulator2` 实例建模，配置入口为 `ddr { ... }`。

关键字段：

- `ddr.enabled`
- `ddr.freq`
- `ddr.channels`
- `ddr.req_size`
- `ddr.nbl`
- `ddr.size_gb`
- `ddr.address_base`
- `ddr.config_path`

当前样例：

- `enabled = true`
- `freq = 1600 MHz`
- `channels = 4`
- `req_size = 32 B`
- `size_gb = 16`
- `config_path = ../configs/ramulator2_configs/DDR4.yaml`

说明：

- `DDR` 不是替代 `HBM`，而是新增的一层较慢、较大容量的主存层
- 当前 `DDR` 由 `StorageController` 管理，并与 `HBM` / `SSD` 共用统一路由框架
- NPU 的 memory-side port 仍以 `HBM channels` 为主，`DDR` 更多体现为远端介质和可显式放置的数据驻留区

## 5. SSD 配置

`SSD` 配置入口为 `ssd { ... }`。

关键字段：

- `ssd.enabled`
- `ssd.address_base`
- `ssd.capacity_bytes`
- `ssd.nchs`
- `ssd.luns_per_ch`
- `ssd.secsz`
- `ssd.secs_per_pg`
- `ssd.pgs_per_blk`
- `ssd.blks_per_pl`
- `ssd.pls_per_lun`
- `ssd.pg_rd_lat`
- `ssd.pg_wr_lat`
- `ssd.blk_er_lat`
- `ssd.ch_xfer_lat`

当前样例：

- `address_base = 0x800000000`
- `capacity_bytes = 0x10000000000`
- page size = `secsz * secs_per_pg = 4096 B`

说明：

- 上游 NPU/NoC 仍以 `32 B` 粒度发请求
- `StorageController -> SSD` 边界会把细粒度请求聚合成 `4 KB page` 级访问
- `SSD` 是事件驱动模型，不按每个 NPU 周期细步推进

## 6. 默认存储策略

张量默认放置策略由 `placement { ... }` 控制。

支持的 `policy`：

- `hbm`: 新张量默认放在 `HBM`
- `ddr`: 新张量默认放在 `DDR`
- `ssd`: 新张量默认放在 `SSD`
- `size_threshold`: 小张量放 `HBM`，大张量放 `SSD`

关键字段：

- `placement.policy`
- `placement.ssd_threshold_bytes`

当前主配置默认：

```json
"placement": {
  "policy": "hbm",
  "ssd_threshold_bytes": 1048576
}
```

对应行为：

- 默认新张量分配到 `HBM`
- 若将 `policy` 改成 `ddr`，则 trace/模型中新建张量默认从 `DDR` 地址段分配
- 若改成 `ssd`，则默认直接从 `SSD` 地址段分配
- 若改成 `size_threshold`，则小于阈值的张量放 `HBM`，大于等于阈值的张量放 `SSD`

## 7. 如何调整

常见调整方式如下。

### 7.1 切换默认驻留介质

修改 `placement.policy`：

```json
"placement": {
  "policy": "ddr",
  "ssd_threshold_bytes": 1048576
}
```

当前仓库已提供三套可直接运行的默认放置配置：

- [configs/systolic_ws_128x128_c4_simple_noc_tpuv4_half_ramulator2_hbm_default.json](/home/yihan.lu/GR_simulator/configs/systolic_ws_128x128_c4_simple_noc_tpuv4_half_ramulator2_hbm_default.json)
- [configs/systolic_ws_128x128_c4_simple_noc_tpuv4_half_ramulator2_ddr_default.json](/home/yihan.lu/GR_simulator/configs/systolic_ws_128x128_c4_simple_noc_tpuv4_half_ramulator2_ddr_default.json)
- [configs/systolic_ws_128x128_c4_simple_noc_tpuv4_half_ramulator2_ssd_default.json](/home/yihan.lu/GR_simulator/configs/systolic_ws_128x128_c4_simple_noc_tpuv4_half_ramulator2_ssd_default.json)

### 7.2 调整 HBM/DDR 容量或地址边界

修改：

- `hbm.size_gb`
- `ddr.size_gb`
- `hbm.address_base`
- `ddr.address_base`
- `ssd.address_base`

要求：

- 三个地址区间不能重叠
- `ssd.address_base` 必须大于 `HBM` 和 `DDR` 的结束地址

### 7.3 调整带宽与并行度

修改：

- `hbm.freq`, `hbm.channels`
- `ddr.freq`, `ddr.channels`
- `ssd.nchs`, `ssd.luns_per_ch`

影响：

- `HBM` 的 `channels` 还会影响 NPU memory port 数量
- `DDR` 主要影响远端主存服务能力
- `SSD` 的通道/LUN/时延参数影响 page 级访问延迟和吞吐

### 7.4 调整请求粒度

修改：

- `hbm.req_size`
- `ddr.req_size`

建议：

- 第一优先保持与 NPU 当前细粒度接口一致，即 `32 B`
- 若改动该值，要同步关注 `core`、`instruction size`、`StorageController` 聚合行为以及已有 trace 是否仍成立

## 8. mem_bench 配置

`mem_bench` 使用 `configs/mem_benchmark_default.json`。

当前已支持：

- `media = ["hbm", "ddr", "ssd"]`

结果默认输出到：

- `results/hbmddrssd/mem_bench`

可直接运行：

```bash
./build/bin/Simulator \
  --config ./configs/systolic_ws_128x128_c4_simple_noc_tpuv4_half_ramulator2_ssd.json \
  --mode mem_bench \
  --bench_config ./configs/mem_benchmark_default.json \
  --bench_output_dir ./results/hbmddrssd/mem_bench
```

## 9. 当前边界

当前实现需要注意以下边界：

- `HBM/DDR/SSD` 地址空间和默认放置已支持
- `HBM` 与 `DDR` 都使用同一份 `ramulator2` 源码，但以不同 YAML 和不同实例运行
- `SSD` 仍通过 page 聚合和事件时间推进
- 上层算子性能模型仍主要按 `HBM` 路径估算，不代表完整的分层数据放置优化已经完成
- 更复杂的驻留管理、自动换入换出策略和 user-level working set 调度仍是后续工作
