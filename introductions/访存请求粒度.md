# 访存请求粒度

本文整理当前仿真器中“访存请求粒度”相关的设计与实现，目标是回答以下问题：

- NPU `operation`、`core`、`interconnect`、`DRAM`、`SSD` 分别以什么粒度看待请求。
- 一个 DRAM 请求从算子生成到返回 `core` 的完整生命周期是什么。
- SSD 为什么不能直接沿用 DRAM 的 `32B` 粒度，当前系统是如何在 `StorageController -> SSD` 边界处理请求合并的。
- 当前实现与 FEMU blackbox 原生 host request 语义相比，有哪些对齐之处，又有哪些额外策略。

为了便于后续查代码，文中会直接引用关键文件路径。

---

## 1. 总览：当前系统里的“粒度”不是一个值，而是分层定义

当前仿真系统并不是从上到下都使用同一种请求粒度，而是分层处理：

1. `Operation` 层：
   - 以“算子输出块 / tile 内数据块”的视角组织访存。
   - 生成的不是单个大字节请求，而是带有许多外存地址项的 `MOVIN/MOVOUT` 指令。

2. `Core` 层：
   - 真正把 `MOVIN/MOVOUT` 展开成很多个 `MemoryAccess`。
   - 这里的原子请求粒度固定是 `dram_req_size`，当前配置为 `32B`。

3. `ICNT` / NoC 层：
   - 把每个 `MemoryAccess` 当成一个 network packet。
   - 所以 NoC 看到的也是 `32B` 请求。

4. `DRAM` 层：
   - 当前 `DramRamulator2` 强制要求每个请求大小必须等于 `dram_req_size`。
   - 所以 DRAM 侧原子粒度也是 `32B`。

5. `SSD` 层：
   - SSD 内部的 FTL/NAND 语义是 page-level。
   - 当前 page 大小为 `4096B`。
   - 因此不能直接把 NPU 发出的所有 `32B` 请求原样送入 SSD；需要在 `StorageController -> SSD` 边界进行组装。

可以把它理解为：

- `NPU / SRAM / NoC / DRAM` 这一侧是 **cacheline-like 细粒度**
- `SSD FTL / NAND` 这一侧是 **page-like 粗粒度**

因此当前系统的关键点不是“选一个统一粒度”，而是：

- **在什么位置把细粒度请求转换成 SSD 能接受的 host-style request**

## 2. 核心数据结构：`MemoryAccess` 与 `Instruction`

### 2.1 `MemoryAccess`

统一的访存请求结构定义在 [src/Common.h](../src/Common.h)：

- `dram_address`
- `spad_address`
- `size`
- `write`
- `request`
- `core_id`
- `issue_time_ps / mem_enter_time_ps / mem_finish_time_ps / return_time_ps`
- `logical_size_bytes`
- `macro_request_id`
- `source_medium / target_medium / destination_medium`
- `controller_generated`
- `ssd_host_request`

其中和“粒度”最相关的字段是：

- `size`
  - 这条 `MemoryAccess` 自己表示的请求大小。
- `logical_size_bytes`
  - 聚合请求或迁移请求使用的逻辑字节数。
- `ssd_host_request`
  - 是否已经是“发往 SSD 的 host-style request”。
- `controller_generated`
  - 是否由 `StorageController` 内部生成，而不是由 `core` 直接发出。

### 2.2 `Instruction`

`Instruction` 同样定义在 [src/Common.h](../src/Common.h)，关键字段有：

- `opcode`
- `dest_addr`
- `size`
- `src_addrs`
- `base_addr`

注意：

- `Instruction::size` 不是“发往 DRAM/SSD 的字节数”
- 它更多是这条指令要在片上 SRAM 里分配多少个 `dram_req_size` 大小的数据块
- 真正的外存请求是由 `src_addrs` 里的每一项在 `Core` 层展开出来的

## 3. NPU operation 层的粒度

### 3.1 operation 不直接生成“大字节请求”

以 GEMM 为例，`MOVOUT` 生成逻辑在 [src/operations/GemmWS.cc](../src/operations/GemmWS.cc)。

在 `MOVOUT` 阶段：

- 算子遍历输出 tile
- 计算该 tile 对应的所有外存输出地址
- 把这些地址放入 `std::set<addr_type> output_set`
- 再构造一条 `Instruction`

关键点：

- `src_addrs = std::vector<addr_type>(output_set.begin(), output_set.end())`
- 也就是说，一条 `MOVOUT` 本质上是：
  - 一个片上源地址 `dest_addr = out_sp_addr`
  - 加上一批片外目标地址 `src_addrs`

同理，`MOVIN` 也是同一个思路：

- 一条 `MOVIN` 不是一个“tensor 大小”的单请求
- 而是“这次 tile 需要读哪些外存地址块”的集合

因此 operation 层的“粒度”可以理解为：

- **以 tile/block 为单位生成访存地址集合**
- 但还没有变成真正的 memory-side 原子请求

### 3.2 为什么 operation 层不直接决定 DRAM/SSD 粒度

因为 operation 层并不知道目标介质，只知道地址。

例如：

- `output_addr` 可能落在 DRAM 区间
- 也可能因为 `place_threshold_bytes` 策略落在 SSD 区间

地址空间归属是在后面由 `StorageController` 判断的，而不是 operation 层。

## 4. NPU core 层的粒度：真正的 `32B` 原子请求在这里出现

### 4.1 `MOVIN`

`Core::handle_ld_inst_queue()` 在 [src/Core.cc](../src/Core.cc)。

当 `opcode == MOVIN` 时：

1. 先在目标 SRAM 上调用 `prefetch(...)`
2. 然后遍历 `front->src_addrs`
3. 对每个地址 new 一条 `MemoryAccess`

生成的字段中，最关键的是：

- `.size = _config.dram_req_size`
- 当前配置里 `dram_req_size = 32`

所以：

- 每个 `src_addrs` 地址项都会变成一条 `32B` 的读请求

### 4.2 `MOVOUT`

`Core::handle_st_inst_queue()` 也是同样的逻辑：

1. 检查对应 SRAM 命中
2. 遍历 `front->src_addrs`
3. 为每个地址生成一条写请求

同样：

- `.size = _config.dram_req_size`
- 即 `32B`

并且每发出一条写请求都会：

- `_waiting_write_reqs++`

这点很重要，因为后面聚合后的 SSD 完成响应必须 fan-out 回原始 waiter，才能把这个计数正确减回去。

### 4.3 core 层粒度结论

真正的 memory-side 原子粒度是在 core 层固定下来的：

- **所有普通 NPU 访存请求都先变成 `32B MemoryAccess`**

这也是当前系统中 DRAM/NoC/SSD 统一的“上游接口粒度”。

---

## 5. ICNT / NoC 层的粒度

在 [src/Simulator.cc](../src/Simulator.cc) 的 ICNT 部分：

1. core 从 `_request_queue` 取出一个 `MemoryAccess`
2. 把它 push 到 interconnect
3. 再由 memory-side node 送给 `StorageController`

因此 NoC 看到的请求仍然是：

- 一个 `MemoryAccess`
- 大小为 `32B`

这意味着：

- 当前 SSD 请求聚合并没有改动 NPU -> NoC 这段建模
- NoC 负载仍然按 `32B` 流量计算

这是当前方案一个很重要的优点：  
**SSD 侧聚合没有破坏 NPU/NoC/DRAM 这条原有的细粒度行为。**

---

## 6. DRAM 的请求粒度

### 6.1 当前 DRAM 粒度就是 `dram_req_size = 32B`

DRAM 侧关键实现是 [src/Dram.cc](../src/Dram.cc)。

在 `DramRamulator2::push()` 中：

- wrapper 会要求请求大小等于 `_config.dram_req_size`

也就是说，当前 DRAM 路径明确假定：

- 一个 DRAM 请求原子大小就是 `32B`

### 6.2 channel 选择也是按 `32B` 粒度地址做的

`Dram::get_channel_id()` 会使用：

- `access->dram_address / _config.dram_req_size`

也就是：

- channel 哈希是按 `32B` 原子块编号做的

因此 DRAM 粒度不只是“请求大小”，还体现在：

- channel mapping
- NoC packet 数量
- SRAM fill 次数

## 7. 一个 DRAM 请求完整生命周期

这里以 `MOVIN` 从 DRAM 读数据到 NPU 为例。

### 7.1 operation 生成 `MOVIN`

例如 GEMM / Conv / Softmax 等 operation 在各自 `initialize_instructions()` 里生成：

- `opcode = MOVIN`
- `src_addrs = 一批外存地址`
- `dest_addr = 片上 SRAM 地址`

### 7.2 core 将其展开为多个 `32B MemoryAccess`

`Core::handle_ld_inst_queue()`：

1. `prefetch(dest_addr, size, size)`
2. 对每个 `src_addrs` 项生成：
   - `dram_address`
   - `spad_address`
   - `size = 32`
   - `write = false`
   - `request = true`
3. push 到 `_request_queue`

### 7.3 Simulator 将请求送入 ICNT

`Simulator::run_simulator()` 的 ICNT 部分：

1. `core->top_memory_request()`
2. `_icnt->push(...)`
3. 从 memory-side node 取出请求

### 7.4 `StorageController` 判断目标介质

`StorageController::dispatch_request()`：

1. 看 `_ssd->owns_address(request->dram_address)`
2. 若地址不在 SSD 区间，则目标介质为 DRAM
3. `route_to_device(...)`

### 7.5 DRAM wrapper 接收请求

`route_to_device()` 对 DRAM 路径：

1. 计算 channel
2. `_dram->push(channel, request)`

### 7.6 Ramulator2 完成后返回原始 `MemoryAccess`

DRAM 完成时：

1. `StorageController::drain_dram_responses()`
2. `handle_completed_access(now_ps, response)`
3. 非 controller-generated 请求直接进 `_ready_responses`

### 7.7 ICNT 把响应送回 core

`Simulator` 再把 `StorageController` 的 ready response：

1. push 回 interconnect
2. `core->push_memory_response(response)`

### 7.8 core 用每个 `32B` 响应填 SRAM

对于读响应：

- `Core::push_memory_response()` 调用 `_spad.fill(...)` 或 `_acc_spad.fill(...)`

SRAM 内部按 `remain_req_count` 统计：

- 只有所有 `32B` 子请求都返回后，该片上目标块才算 ready

### 7.9 DRAM 生命周期总结

DRAM 请求完整函数调用流可简写为：

`Operation::initialize_instructions()`
-> `Core::handle_ld_inst_queue()/handle_st_inst_queue()`
-> `Simulator::run_simulator()` ICNT push
-> `StorageController::dispatch_request()`
-> `Dram::push() / DramRamulator2::push()`
-> `StorageController::drain_dram_responses()`
-> `Simulator` ICNT return path
-> `Core::push_memory_response()`

其中真正的 DRAM 原子请求粒度始终是：

- **`32B`**

## 8. SSD 的原生粒度：page-level，而不是 `32B`

### 8.1 SSD 内部 page 大小

当前 SSD page 大小来自配置：

- `secsz = 512`
- `secs_per_pg = 8`

因此：

- `page_bytes = 4096B`

### 8.2 SSD 内部 FTL/NAND 处理语义

当前 `src/Ssd.cc` 是对 FEMU blackbox 核心行为的嵌入式复现。

写路径：

- `process_write()`
- `slba/nlb -> start_lpn/end_lpn`
- 对每个 `lpn`：
  - 旧页失效
  - 分配新 page
  - 更新 `maptbl/rmap`
  - `ssd_advance_status(... NAND_WRITE ...)`

读路径：

- `process_read()`
- 同样按 `lpn` 查映射并调用 `ssd_advance_status(... NAND_READ ...)`

也就是说，SSD 内部天然是：

- **host request 入口**
- **page/LPN 展开**
- **NAND page 时延**

而不是：

- `32B cacheline` 直接进 NAND

---

## 9. 为什么 SSD 不能直接沿用 DRAM 的 `32B`

如果把 core 发出的每个 `32B` 写直接当成独立 SSD host write：

- 每个小写都可能触发一次 page-level `process_write()`
- 同一逻辑 page 会被重复 program 很多次
- 写次数和尾延迟都会严重放大

旧版系统确实存在这个问题，所以后来引入了 SSD 请求合并。

但引入合并后，还要解决另一个问题：

- 不能为了“合得更多”而完全变成一个新的 SSD page cache 模型
- 否则就偏离了 FEMU blackbox 的原生语义

因此当前实现选择了折中：

- 保留上游 `32B`
- 不改 `Ssd` 内部 FEMU 风格 FTL/NAND
- 只在 `StorageController -> SSD` 边界做 host-request 组装

---

## 10. 当前 SSD 请求粒度与合并策略

### 10.1 总体思路

当前 SSD 路径分成两层：

1. 上游：
   - core 仍然发 `32B MemoryAccess`
2. SSD 边界：
   - `StorageController` 把这些 `32B` 请求组装成 page-sized SSD host request
3. SSD 内部：
   - `Ssd::push()` 接收 `ssd_host_request=true` 的请求
   - 按当前 FEMU 风格 FTL/NAND 处理

### 10.2 读：in-flight merge

读路径在 `StorageController::handle_ssd_read()`：

1. 按 `ssd_page_addr()` 对齐到 page
2. 如果该 page 已有 in-flight read：
   - 把 waiter 挂到 `_ssd_read_aggregates`
3. 否则：
   - 创建一个 page-sized 聚合父请求
   - 设置 `ssd_host_request = true`
   - 直接送进 SSD

完成后：

- `complete_ssd_aggregate()` 再把原始 waiters fan-out 回上游

因此读的策略是：

- **首个请求立即发起 page read**
- **后续同页请求复用这次在途 page read**

它不是 page cache，因为：

- 只复用“在途请求”
- 读完成后不保留 resident page

### 10.3 写：controller 边界聚合

写路径在 `StorageController::handle_ssd_write()`。

当前策略是：

- 同一写流允许同时保留多个 page
- 但最多只保留 **4 个活跃 page**
- 如果将要开第 5 个 page：
  - 先 flush 该写流里最老的那个 page
  - 再接纳新 page

同一个 pending page 在以下条件下会 flush：

- page 内聚合的 `32B` 数量达到 `4096/32=128`
- 超过 idle timeout
- 被“活跃 page 上限”策略挤出

flush 时：

1. 生成一个 page-sized聚合父请求
2. `ssd_host_request = true`
3. 送入 SSD
4. SSD 完成后，再把原始 waiter fan-out 回去

### 10.4 为什么响应数量不会变少

这是当前设计里非常重要的一点：

- SSD 侧 host request 数会减少
- 但 controller 会在完成时把父请求 fan-out 回原始 `32B` waiter

因此对 core 看来：

- 写响应数量仍然和原先发出的 `32B` 请求数一致
- `_waiting_write_reqs` 仍然逐条 decrement

这保证了：

- 上层 NPU 管理逻辑不需要改
- 只是在 SSD 边界改变了请求的“下行形态”

## 11. 当前 SSD 请求策略与 FEMU 原生 blackbox 的关系

### 11.1 对齐之处

当前实现和 FEMU blackbox 对齐的关键点在于：

1. SSD 内部仍然按 host request 的 `slba/nlb` 语义工作
2. `process_read()/process_write()` 仍然是 page-level FTL
3. `ssd_advance_status()` 仍然负责 NAND read/write/erase 时延
4. 不改原有 `maptbl/rmap/GC/write pointer` 语义

也就是说：

- **改的是“送进 SSD 的请求形态”**
- **没改 SSD 内部 FTL/NAND 行为**

### 11.2 与 FEMU 原生不同之处

FEMU blackbox 原生前端是：

- QEMU/NVMe
- 主机本来提交的就是块设备请求
- 然后 FTL 再按 page 展开

而本系统上游是：

- NPU core
- 先发出很多 `32B` cacheline 请求

因此当前 controller 侧聚合其实是在补一层：

- 把 NPU `32B` 请求转换成更像 FEMU host request 的形态

### 11.3 当前实现比 FEMU 更强的地方

严格说，当前“每个写流最多 4 个活跃 page”的策略，已经比 FEMU blackbox 原生更强：

- FEMU 原版没有显式 page 写缓冲
- 它不会在 FTL 前再维持一个“多 page 活跃写缓冲”

因此当前设计的定位应理解为：

- **保留 FEMU 的内部 FTL/NAND 精度**
- **在 controller 边界增加一层受限的 host-request 组装**

这在系统集成上是合理的，但要明确它不是 FEMU 原版的字面复制。

## 12. 其他相关信息

### 12.1 张量放置策略与“为什么会走 SSD”

张量是否落在 SSD 地址空间，不由 operation 决定，而由全局地址分配策略决定：

- `place_threshold_bytes`
- 大于等于阈值的 tensor 分配到 SSD 地址空间

因此：

- 同一条 `MOVIN/MOVOUT`
- 只要目标地址落在 SSD 区间
- 就会自动进入 SSD 路径

### 12.2 trace 前端的 SSD 预填充

为了让放在 SSD 上的输入/权重后续真的能“读到已存在数据”，当前 trace 前端增加了 `prefill_ssd_tensors()`：

- `TraceModel` 在模型初始化后扫描位于 SSD 地址空间的 tensor
- 直接调用 `Ssd::prefill_range()`
- 在 FTL 中建立有效映射
- 不计入仿真总时间

否则会出现：

- 地址在 SSD
- 但读时没有映射，导致没有真实 NAND read 行为

### 12.3 `dram<->ssd migration` 也走同一套粒度逻辑

`StorageController` 原本就支持：

- `DRAM -> SSD`
- `SSD -> DRAM`

的纯存储侧迁移。

当前改造后：

- 迁移读 SSD 时也会使用 read in-flight merge
- 迁移写 SSD 时也会使用 controller 边界写聚合

这使得 SSD 粒度策略不只覆盖 operation 写回，也覆盖 migration。

### 12.4 当前策略的性能含义

从实验上看：

- 完全保守的单 page 聚合，会导致大量 page write、长尾严重
- 完全放开多活跃 page，会显著降低写次数，但偏离 FEMU 太多
- 当前采用“最多 4 个活跃 page”的折中：
  - 明显降低长尾
  - 又不会像无限 page buffer 那样过于乐观

因此当前策略更接近：

- **一个有限深度的 controller-side SSD host request assembler**

而不是：

- 完整 SSD page cache
- 或完全不做组装

---

## 13. 最终总结

当前 ONNXim 集成 SSD 后，请求粒度问题的核心可以概括为三句话：

1. **NPU/DRAM 侧仍然坚持 `32B` 原子请求。**
   - operation 生成地址集合
   - core 真正把它展开成 `32B MemoryAccess`
   - NoC 和 DRAM 都按这个粒度工作

2. **SSD 内部仍然坚持 page-level FTL/NAND 语义。**
   - page 大小 `4KB`
   - 读写最终按 `LPN/page` 展开
   - 内部行为尽量对齐 FEMU blackbox

3. **两者之间的粒度转换发生在 `StorageController -> SSD` 边界。**
   - 读：同页 in-flight merge
   - 写：有限深度的多 page 聚合
   - 完成时再 fan-out 回原始 `32B` waiter

所以，当前仿真器处理“请求粒度”的整体思路不是：

- 把所有层都改成同一个粒度

而是：

- **在不同层保留最适合该层的粒度**
- **只在 SSD 边界做受控的语义转换**

这也是 ONNXim 集成 SSD 后，为了同时兼顾：

- NPU 细粒度内存行为
- SSD page-level FTL/NAND 行为
- 与 FEMU blackbox 对齐程度
- 系统级可运行性与性能

所采用的核心设计。
